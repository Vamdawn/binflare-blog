# 个人博客 MVP 设计（2026-03-11）

## 1. 背景与目标

- 业务目标：在 1 周内上线个人博客 MVP，并支持稳定周更。
- 成功标准：以作者发布稳定性优先，而非一次性功能完整度。
- 确认范围：首版仅包含文章列表页与文章详情页。

## 2. MVP 范围

### 2.1 包含内容

1. 基于 Markdown 的文章内容管理（Git 提交触发发布）
2. 文章列表页（按日期倒序）
3. 文章详情页（Markdown 正文渲染）
4. 基础 SEO（页面 title、description、sitemap）
5. 构建期内容校验（frontmatter、slug 唯一性）

### 2.2 明确不做（YAGNI）

1. 后台管理系统
2. 评论、点赞、搜索、标签分类
3. 文章数据库化与动态文章 API

## 3. 方案对比与结论

### 3.1 备选方案

1. 纯静态渲染（构建期读取 Markdown）
2. 服务端读取 Markdown 并提供 API
3. Markdown 入库后由数据库驱动前端

### 3.2 结论

采用方案 1：纯静态渲染。

理由：

- 与“快速稳定发布”目标一致，依赖少、故障面最小。
- 1 周上线风险最低，部署流程简单。
- 后续需要扩展时可再引入 API 或数据库层。

## 4. 信息架构与页面设计

### 4.1 页面结构

1. `/`：文章列表页
2. `/posts/:slug`：文章详情页

### 4.2 组件最小集合

1. `SiteLayout`：统一站点布局容器
2. `PostCard`：列表项（标题、日期、摘要、链接）
3. `MarkdownRenderer`：详情正文渲染
4. `SeoHead`：每页基础 SEO 注入

## 5. 内容模型与 frontmatter 规范

### 5.1 frontmatter 字段（定稿）

1. `title: string`（必填）
2. `date: string`（必填，格式 `YYYY-MM-DD`）
3. `summary?: string`（选填）
4. `draft?: boolean`（选填，默认 `false`）
5. `slug` 不写入 frontmatter，始终由文件名生成

示例：

```md
---
title: "我为什么开始写这个博客"
date: "2026-03-11"
summary: "从零搭建到稳定周更的第一周记录。"
draft: false
---

正文...
```

### 5.2 校验规则

1. 缺少 `title` 或 `date`：构建失败
2. `date` 格式不合法：构建失败
3. `slug` 冲突：构建失败
4. 生产构建自动排除 `draft: true`

## 6. 数据流与发布流

### 6.1 写作到上线

1. 在 `content/posts/*.md` 编写文章
2. 提交并推送到仓库
3. CI 执行构建与内容校验
4. 构建成功后部署静态站点

### 6.2 运行时读取

1. 列表页读取构建产物中的文章元数据
2. 详情页通过 `slug` 匹配文章内容
3. 未命中 `slug` 返回 404 页面

## 7. 错误处理策略

1. 构建期优先：内容问题在 CI 直接失败并阻断发布
2. 运行期兜底：详情页无匹配文章时进入 404
3. 摘要兜底：`summary` 缺失时可由正文前 N 字自动生成

## 8. 测试策略（MVP 最小集合）

1. 解析测试：Markdown/frontmatter 合法与非法样例
2. 排序测试：列表按 `date` 倒序
3. 过滤测试：生产环境不包含 `draft: true`
4. 路由测试：有效 `slug` 可渲染、无效 `slug` 返回 404

## 9. 验收标准

1. 单篇文章从编写到上线可在 10 分钟内完成
2. 列表页稳定展示最新文章并按日期倒序
3. 详情页稳定渲染 Markdown 正文
4. `draft` 文章不会出现在生产环境
5. 内容不合规会在构建阶段给出明确失败信息

## 10. 上线后一周观察指标

1. 发布次数：目标 `>= 1 / 周`
2. 发布失败次数：目标 `0`
3. 单次发布平均耗时：目标 `<= 10 分钟`

## 11. 后续边界

本设计仅覆盖 MVP。若后续迭代，优先顺序建议为：

1. 基础 SEO 完善（Open Graph、RSS）
2. 标签与归档
3. 搜索能力
4. 统计与订阅
